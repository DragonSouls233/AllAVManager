# MDCX 项目开发规则（AI 必须读取）

> 版本：v1.1
> 适用范围：所有 AI 开发的代码修改、新增、重构任务
> 核心原则：**优先复用成熟代码，最小化自研成本**

---

## 一、代码引用优先级

### 1.1 三级参考代码库

`G:\MDCX\.references\` 目录下的代码资产按成熟度分为三级：

| 级别 | 目录 | 来源 | 成熟度 | 适用场景 |
|------|------|------|--------|----------|
| **P0** | `GitHub\*` | 行业已验证的开源项目 | 高（生产级） | **默认首选** |
| **P1** | `本地\*` | 自研原型项目（VaultX/ReelSorter/PornSimilarityPlatform）| 中（架构参考） | 仅参考架构设计和模块拆分方案 |
| **P2** | `特殊项目\*` | 自研已验证的功能模块 | 中（已验证） | 特定功能的完整方案（海角/下载器/查重） |

### 1.2 选择优先级规则

```
IF  功能在 GitHub 目录有成熟实现 → 直接复用/适配 (P0)
ELIF 功能在 特殊项目 目录有已验证实现 → 按 GitHub 规范重构后复用 (P2)
ELIF 功能在 本地目录 有参考实现 → 提取架构设计思路，按 GitHub 规范重写 (P1)
ELSE → 自研开发，对齐 GitHub 代码规范
```

### 1.3 禁止行为

- 禁止直接复制 `本地\*` 目录的低成熟度代码到生产代码
- 禁止在 `GitHub\*` 有成熟方案时自研新轮子
- 禁止混合多种实现风格 — 同一功能只能有一种标准化实现

## 三、多方案合并规则

### 3.1 触发条件

当同一功能在 >= 2 个不同级别代码库中存在实现时，必须执行多方案合并。

### 3.2 合并步骤

```
1. 功能拆解：列出各方案的完整功能点清单
2. 差异分析：标记各方案的独有优势特性
3. 冲突识别：找出接口签名、数据格式、错误处理的冲突点
4. 方案设计：取各方案的优势特性，设计统一的标准化接口
5. 兼容层：确保新接口兼容现有调用方（保留旧接口签名）
6. 验证：合并后所有原功能特性必须通过回归测试
```

## 四、代码规范要求

### 4.1 架构分层

```
app/
  api/routes/      # 路由层 - 只做 HTTP 映射和参数校验
  services/        # 服务层 - 核心业务逻辑
  db/              # 数据层 - ORM 模型和数据库操作
  tasks/           # 任务层 - 异步任务和扫描器
  crawlers/        # 爬虫层 - 站点适配器和刮削逻辑
  utils/           # 工具层 - 通用工具函数
  config/          # 配置层 - 配置模型和管理
```

### 4.2 代码风格

- **类型注解**：所有函数参数和返回值必须标注类型
- **异常处理**：使用 try/except 包裹可能失败的 IO/网络操作
- **日志**：使用 app.utils.logger.get_logger(__name__)，不直接用 print
- **配置**：通过 app.config.manager.get_config() 读取，不硬编码
- **异步**：IO 操作用 async/await，CPU 密集型用 run_in_executor

### 4.3 参考来源标注

每个文件头部必须标注参考来源：

```python
"""
功能描述

参考来源：
- P0: mdcx-master/mdcx/llm.py (MIT License)
- P2: 特殊项目/提取/core/extractor.py (自研已验证)
- P1: VaultX/src/shared/... (仅参考架构,无代码复用)

整合说明：
- 生命周期管理: 参考 P0 mdcx
- 提取流程: 参考 P2 提取目录
- 新增: 批量刮削、全局单例
"""
```

如果功能是从某来源直接复制的，需标注来源和许可协议。

### 7.2 PORNHub 模块优化方案

PORNHub 是 5 个模块中唯一薄弱环节，现有爬虫仅 ~100 行基础实现。从参考项目中找到 2 个高价值方案：

| 参考项目 | 方案 | 核心价值 | 整合方式 |
|----------|------|----------|----------|
| **phdownloader-master** (P0) | yt-dlp impersonate | 通过 curl_cffi 模拟 Chrome 绕过 CF 反爬；完整重试策略 | 复用 yt-dlp 调用模式和反爬方案到 Pornhub 爬虫 |
| **PornHubDL-main** (P0) | 页面注入 flashvars | 提取最完整元数据（标题/标签/演员/评分/播放量） | 将 flashvars 提取逻辑集成到爬虫的元数据获取中 |

整合后的 PornHub 爬虫架构：

```
app/crawlers/pornhub.py (增强版)
  - 元数据获取: 参考 PornHubDL-main flashvars 页面注入 (P0)
  - 反爬策略: 参考 phdownloader-main yt-dlp impersonate (P0)
  - 现有: 视图键(viewkey)解析 + 扫描器流程 (已有)
```

### 7.3 下载器增强方案

MDCX 现有 3 种 BT 下载器（Aria2/qB/TR），但缺少**直链/流媒体下载**能力。PornSimilarityPlatform (P1) 提供了完整的补充方案：

| 现有（MDCX） | 待增强（从 P1 引入） |
|-------------|---------------------|
| Aria2 下载器（BT） | **YtDlpEngine**: 1000+ 站点支持、impersonate、CF 绕过 |
| qBittorrent 下载器（BT） | **M3U8Engine**: HLS 流媒体下载、加密流解密 |
| Transmission 下载器（BT） | **HttpEngine**: 多线程分片下载、断点续传 |
| — | **DownloaderFactory**: 多引擎自动检测和回退 |
| — | **DownloadCacheDB**: 下载缓存去重 |
| — | **TempDownloadManager**: 先下缓存后移动 |

整合策略：
```

现有下载器保持不变：
  app/services/aria2_downloader.py    - 沿用
```
